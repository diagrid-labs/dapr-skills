# JSON-Based Intermediate Representation (IR) Generation for Workflow Diagrams

**Objective:** Generate a robust, streamable Intermediate Representation (IR) of a workflow diagram, focusing on explicitly represented elements and direct connections, suitable for subsequent processing and code generation. Implicit structural elements (like certain gateways) will be inferred in a later, deterministic step based on this IR.

**Output Format: JSON Lines**

This prompt generates the IR in JSON Lines format. **Each line of the output MUST be a complete, valid JSON object.** This format is crucial because:

*   **Streamability:** Output can be processed line-by-line, handling large diagrams without exceeding memory or token limits associated with single large JSON objects.
*   **Efficiency:** Each line can be parsed and validated independently.

**Strict Adherence:** Follow the steps precisely. Do *not* output a single large JSON array or object wrapping multiple records. Output a *sequence* of JSON objects, one per line.
*   The output MUST consist **ONLY** of valid JSON objects, one object per line.
*   Do **NOT** wrap the entire output or individual lines in markdown code blocks (e.g., ```json ... ```).
*   Do **NOT** include any introductory text, explanations, comments, or summaries outside of the JSON Line objects themselves.
*   Each line MUST start with `{` and end with `}`.
*   The ONLY exception is the very final `end_of_ir_stream` marker or a final `error` marker if processing fails or is truncated.


**JSON Line Record Types (`__type`):**

Each JSON line object will contain a `__type` attribute indicating the type of record. Possible values:

*   `metadata`: Information about the diagram and the IR generation process. (See Step 1.5)
*   `content_warning`: A warning about prohibited content in the input image. (See Step 0)
*   `system`: **(For Generation Monitoring Only)** Progress updates during IR generation. *Downstream tools consuming this IR for code generation should typically ignore records with `__type: "system"`.*
*   `participant`: A workflow participant/role (e.g., user, system, department). (See Step 2A)
*   `activity`: A task, wait state, or other unit of work in the workflow. (See Step 2B)
*   `gateway`: An *explicitly drawn* decision (exclusive, inclusive, event-based) or synchronization (parallel) point. (See Step 2C)
*   `data_object`: A data input, output, store, or message used in the workflow. (See Step 2D)
*   `failure_handling`: Describes alternative flows triggered by boundary events (error, timer, message, etc.) attached to activities. Represents interruptions and exception paths. (See Step 2E)
*   `unrecognized_item`: Flags elements that couldn't be categorized, require clarification, or indicate potential issues (e.g., missing conditions, ambiguity). (See Step 2F)
*   `start_end_node`: A start or end event of the workflow (e.g., start, end). (See Step 2G)
*   `edge`: A sequence flow connection representing direct control flow between *explicitly identified* elements. May include an optional `visual_label` attribute capturing text seen on the diagram's edge arrow. (See Step 3)
*   `result`: The final validation result *of the explicitly extracted structure* after Step 4. (See Step 4 & 5)
*   `error`: Indicates a fatal error occurred during processing (e.g., diagram type, validation failure, token limit). May include an optional `element_id` referencing the specific IR element causing the error, if applicable. (See relevant Steps & Error Handling)
*   `end_of_ir_stream`: The very last line indicating successful completion and validation of the explicitly extracted IR. (See Step 5)

**Note on Implicit Elements:** This prompt focuses on extracting the *explicit* structure. Implicit gateways (those not explicitly drawn but required by flow logic like joining alternative paths or handling parallel flows) are **NOT** generated by this prompt. They should be inferred deterministically by downstream code analyzing the `edge` connectivity produced here, potentially using hints like `edge.condition_text`.

**Progress Updates (`system` type):**

As the prompt processes each major step, it *may* output JSON Line objects indicating progress. These are primarily for monitoring the generation process.

```json
{"__type": "system", "__step": "<step_number>", "__description": "<step_description>"}
```

*   `__type`: Always "system".
*   `__step`: Current step number (e.g., "1", "2.A", "2B.1", "3").
*   `__description`: Brief description of the current step.

**End Marker (`end_of_ir_stream`):**

After successfully generating *and validating* (Step 4 passes) all *explicit* workflow IR JSON Lines objects, a final marker signals the end of the structured data stream:

```json
{"__type": "end_of_ir_stream", "__description": "End of JSON Lines IR output."}
```

*   This marker will **ONLY** be present if the entire process completes successfully and Step 4 validation (`result.success`) is `true`.
*   Parsers should ignore any text after this marker.

## ⚠️ Important: Output Token Limits

LLM tools may have token limits. If a diagram is complex, the full JSON Lines output might exceed this. Follow these guidelines:

1.  **Prioritize Essential Information:**
    *   **MOST CRITICAL:** `metadata`, `participant`, `activity`, `gateway` (explicit only), `start_end_node`, `edge`, `failure_handling`. Always include `error` if validation fails or truncation occurs, and `end_of_ir_stream` on full success.
    *   **LESS CRITICAL:** `data_object`, `unrecognized_item` (especially those with `impact: low` or `ask_user: false`).
    *   System progress updates (`__type: "system"`) should generally be included but are the first candidates for omission if absolutely necessary *after* simplifying descriptions.

2.  **Be Concise:** Use short names/IDs. Avoid unnecessary words/repetition in descriptions. Omit non-essential optional fields *before* truncating less critical sections.
3.  **Truncation Flag:** If truncation occurs, set `truncated: true` in the `metadata` object. If `metadata` was already output, ensure it reflects this. Output `truncated: false` if complete.
4.  **Error Handling (Token Limit):** If generation stops due to token limits:
    *   Output all valid JSON Lines generated *before* the truncation point.
    *   Ensure `truncated: true` is set in the `metadata` object (if output).
    *   As the **very last line**, output the specific error object:
        ```json
        {"__type": "error", "error": "Output truncated due to token limit. The generated IR is incomplete and should not be considered fully valid.", "truncated": true}
        ```
    *   Do **NOT** output the `end_of_ir_stream` marker.

## 🚀 Step 0: Content Safety Check

**Evaluate the input image based *only* on the "PROHIBITED CONTENT" rules listed in the initial <guardrails> System Prompt.**

*   **If the image CONTAINS prohibited content:**
    1.  Output the following single JSON Line:
        ```json
        {"__type": "content_warning", "status": "offensive", "reason": "Input image contains content violating safety guidelines.", "action_recommended": "do_not_store_image"}
        ```
    2.  **STOP PROCESSING IMMEDIATELY.** Do not proceed to Step 1. Do not output any other lines.

*   **If the image DOES NOT contain prohibited content:**
    1.  Do **NOT** output any `content_warning` line.
    2.  Proceed normally to Step 1.

**Part 2: Steps 1 - 1.5 (Diagram Identification & Metadata)**

```markdown
## 🚀 Step 1: Identify Diagram Type and Assess Workflow Complexity

**NO Output Progress Update for this sub-step.**

**Task:** Classify the input image and assess its complexity as a workflow.

1.  **Analyze:** Determine if the image depicts a step-by-step workflow process (like BPMN, flowchart, sequence diagram focused on process flow).
2.  **Classify:**
    *   **Workflow Diagram:** Identify as `workflow`. Assess complexity: `low`, `medium`, or `high` based on the number of steps, branching, participants, and overall intricacy.
        * Count the number of distinct activities in the diagram.
        * If the activity count exceeds 10, set complexity to `high` and proceed to output a complexity threshold error.
        * If the activity count is 10 or fewer, proceed to Step 1.5.
    *   **Other Diagram Type:** (e.g., pure architecture, class diagram, mind map) Identify the type if possible (e.g., `architecture_diagram`). Set complexity to `unassessed`. Proceed to output error.
    *   **Unclear/Unrecognized:** If it's not clearly identifiable as a workflow or other known type, classify as `unknown_diagram`. Set complexity to `unassessed`. Proceed to output error.

**Important Context:** Focus on *workflow steps* (activities, decisions, flow). Distinguish core workflow elements from purely contextual elements (e.g., external systems shown for context but not actively participating *in the flow logic*). Databases or external services are typically represented as `data_object` interactions within activities, not as `participant`s unless they have an active role defined by a swimlane or clear interaction pattern.

📌 **Error Output (If Not a Workflow or Unassessable):**
If the diagram is not classified as `workflow` or complexity cannot be assessed, output ONLY the following error JSON Line and STOP:

```json
{
  "__type": "error",
  "check_id": "DIAGRAM_TYPE_VALIDATION",
  "diagram_type": "<Actual Identified Type or unknown_diagram>", // LLM's assessment of the diagram type
  "error": "Diagram could not be interpreted as a processable workflow.", // Concise internal reason
  "user_feedback": "<LLM-Generated Specific Feedback>" // Placeholder for LLM output
}
```

📌 **Error Output (If Complexity Exceeds Threshold):**
If the workflow diagram contains more than 10 activities, output ONLY the following error JSON Line and STOP:

```json
{
  "__type": "error",
  "check_id": "DIAGRAM_COMPLEXITY_ABOVE_THRESHOLD",
  "diagram_type": "workflow",
  "complexity": "high",
  "activity_count": "<Actual Number of Activities>", 
  "error": "Workflow exceeds maximum allowed complexity threshold of 10 activities.",
  "user_feedback": "This workflow diagram has too many activities (found <Actual Number>). Please simplify your diagram to 10 or fewer main activities."
}
```

**Instructions for LLM Generating `user_feedback`:**

"If you determine the diagram is not a processable workflow, you MUST generate a helpful `user_feedback` message. This message should:

1.  **State the Conclusion Briefly:** Start with a direct statement like 'Cannot process as a step-by-step workflow because...'
2.  **Identify 1-2 MAIN Reasons (Specific to *this* diagram):** Briefly explain the primary reason(s) it failed. Focus on the most significant issues observed. Examples of *types* of reasons:
    *   '...it appears to be a static diagram (like architecture or an org chart).'
    *   '...the sequence of steps is unclear or arrows are missing/ambiguous.'
    *   '...it uses non-standard shapes extensively for actions or decisions.'
    *   '...it lacks a clear starting or ending point.'
3.  **Provide BRIEF, Targeted Suggestions:** Offer 1-2 concrete visual fixes directly related to the identified reasons. Examples of *types* of suggestions:
    *   'Suggestion: Please represent the process as a sequence of actions over time.'
    *   'Suggestion: Ensure clear arrows connect all steps sequentially.'
    *   'Suggestion: Try using rectangles for steps and diamonds for decisions.'
    *   'Suggestion: Clearly indicate the start and end points of the flow.'
4.  **Be Concise:** Avoid lengthy explanations or listing all possible problems. Focus on the key issues for *this* diagram.
5.  **Avoid Internal Jargon:** No `IR`, `schema`, etc.
6.  **CRITICAL FORMATTING:** The entire message MUST be a single JSON string value, using escaped newlines (`\\n`) for any necessary line breaks within the text. Do NOT output literal newlines.

---

**Example of Concise, Correctly Formatted Output (Applying the new instructions):**

```json
{"__type": "error", "check_id": "DIAGRAM_TYPE_VALIDATION", "diagram_type": "entity_relationship_diagram", "error": "Diagram could not be interpreted as a processable workflow.", "user_feedback": "Cannot process as a step-by-step workflow because it appears to show entity relationships rather than a sequence of actions.\\nSuggestion: To process, redraw using shapes like rectangles for steps, connected by arrows showing the flow over time from a start to an end."}
```

**Another Example (Different Reason):**

```json
{"__type": "error", "check_id": "DIAGRAM_TYPE_VALIDATION", "diagram_type": "unknown_diagram", "error": "Diagram could not be interpreted as a processable workflow.", "user_feedback": "Cannot process as a step-by-step workflow because the sequence flow is unclear due to missing or ambiguous connection arrows.\\nSuggestion: Ensure all steps are connected sequentially with clear arrows indicating the process direction."}
```

*   Do NOT output `result` or `end_of_ir_stream`.

## 🚀 Step 1.5: Determine Workflow Name/Description and Create Metadata Record

**NO Output Progress Update for this sub-step.**

**Task:** Define essential metadata *after* confirming it's a workflow in Step 1.

1.  **Determine Name & Description:**
    *   **Name:** Infer a concise `PascalCase` name suitable for a class or type definition (e.g., `OrderProcessing`, `ClaimHandling`). Look for titles or infer from the main goal.
    *   **Description:** Write a brief sentence describing the overall purpose or goal of the workflow.
2.  **Output `metadata` Record:** Output the metadata as the **first JSON Line** of the IR stream (after potential Step 0 warning).
    *   Use `diagram_type`, `complexity` from Step 1.
    *   Use inferred `name` and `description`.
    *   Set `schema_version` to `"1.0"`
    *   Set `truncated` to `false` initially (may be updated later if token limit hit).
    *   Set `feedback_required` to `false` initially (will be set to `true` in Step 2F if any `unrecognized_item` requires user input).

```json
// Example Metadata Output:
{"__type": "metadata", "schema_version": "1.0", "diagram_type": "workflow", "complexity": "medium", "truncated": false, "name": "OrderProcessingWorkflow", "description": "Handles the end-to-end process for customer orders.", "feedback_required": false}
```

**Part 3: Steps 2A - 2C (Participants, Activities, Explicit Gateways)**

```markdown
## 📌 Step 2A: Identify Participants

**Output Progress Update:**
```json
{"__type": "system", "__step": "2.A", "__description": "Identify Participants"}
```

**Task:** Identify active participants (roles, systems, departments) *performing* actions *within* the workflow logic.

**Clues (Prioritize swimlanes):**

*   **Swimlanes/Pools:** Each lane usually represents one participant. THIS IS THE STRONGEST INDICATOR.
*   **Visual Grouping/Color:** Elements grouped or colored similarly might belong to the same participant.
*   **Labels Near Activities:** Text indicating roles (e.g., "Customer," "Admin," "Backend System") near task boxes, especially if no swimlanes exist.
*   **Connections:** Who initiates or performs specific activities.
*   **Default:** If no clues, assign a default like `"System"` or `"User"`, but assign low confidence.

**Distinguish Participants vs. External Context:** Participants are *active* performers in the flow. External systems/databases interacted *with* are typically handled via `data_object` interactions within activities, not as participants themselves, unless explicitly modeled with a lane/active role.

**Output:** One JSON Line per participant identified.

```json
// Example Participant Output:
{"__type": "participant", "id": "customer", "role": "Customer", "confidence": "high"}
{"__type": "participant", "id": "order_system", "role": "Order Processing System", "confidence": "high"}
{"__type": "participant", "id": "manager_1", "role": "Manager1", "confidence": "medium"}
{"__type": "participant", "id": "manager_2", "role": "Manager2", "confidence": "medium"}

// Example when no participants are found:
{"__type": "participant", "id": "system", "role": "System", "confidence": "low"}
```

---

## 📌 Step 2B: Extract Activities

**Output Progress Update:**
```json
{"__type": "system", "__step": "2.B", "__description": "Extract Activities"}
```

**Task:** Identify and describe all distinct work units, tasks, or wait states shown in the diagram. Systematically scan the diagram for activity shapes (typically rectangles) and intermediate event symbols that function as standalone wait states (typically circles with an icon representing the event type, like a timer, message, signal, etc.).

*   **Progress Update per Participant (Optional but Recommended):**
    ```json
    {"__type": "system", "__step": "2B.1", "__description": "Extract Activities for Participant: <participant_id>"}
    ```

**Rules:**

*   **Extract ALL Visible/Implied Actions:** Systematically scan the entire diagram. Do NOT invent activities. Represent tasks explicitly shown (e.g., rectangles) or clearly implied wait states (e.g., intermediate timer/message catch events, circles with an icon representing the event type, like a timer, message, signal, etc.). **Crucially, ensure that any element visually represented as an activity is output as an `activity` record.**
*   **Handle Dashed/Dotted Line Interactions:** If an activity visually interacts with another element (especially external systems or participants) via dashed or dotted lines (often labeled 'message', 'API call', 'pub/sub'):
    *   Assign an appropriate `task_type` to the activity reflecting the interaction (e.g., `send_task` for outgoing, `service_task` for request/response, `receive_task` or `wait_for_event` for incoming/waiting).
    *   The dashed/dotted line itself is *not* sequence flow and should *not* be generated as an `edge` in Step 3. Optionally, create an `unrecognized_item` (Step 2F) with `category: "annotation"` to describe the visual non-sequence flow line if contextually important.
*   **Check References:** Pay close attention to elements referenced as targets (`next` field) in gateway flows (Step 2C) and as destinations (`to` field) in edges (Step 3). If such a target element looks like an activity in the diagram, ensure its corresponding `activity` record is generated in this step.
*   **Intermediate Catching Events (Following Event-Based Gateway):** Do **NOT** create separate `activity` nodes for intermediate catching events (message, timer, signal, etc.) that *immediately follow* an Event-Based Gateway. These events are defined within the gateway's `flows` array (Step 2C).
*   **Standalone Intermediate Catching Events:** Create an `activity` with `type: "wait_for_event"` ONLY for intermediate catch events that act as standalone wait states in the flow (i.e., *not* immediately following an event-based gateway). This includes intermediate timer events (duration or specific time) and message/signal catch events.
*   **Boundary Events:** Events attached to the border of an activity (timer, error, message, conditional, signal, etc.) represent interruptions or alternative paths. **Do NOT modify the main activity record for these.** Instead, create separate `failure_handling` records for them in Step 2E.
*   **Multi-Instance Markers:** Check for standard BPMN multi-instance markers on the activity shape itself.
*   **Start Events with Triggers as Activities:** If a visual element that typically represents a Start Event (e.g., a circle with a thin border, usually the first element in a flow) contains an icon indicating a specific trigger (e.g., an envelope for a message start event, a clock for a timer start event), this element **MUST be generated as an `activity` record with `type: "wait_for_event"` in this step (Step 2B).** This activity serves as the initiating point of the workflow.
*   *   Its `description` should reflect its role as the workflow trigger (e.g., "Waits for the initial trigger message to start the process.").
*   *   Its `participant` field should be assigned (e.g., to a default system participant if no other is indicated).
*   *   Its `events` array must contain one entry describing the specific trigger. For example, for a message start event:
        ```json
        // Inside the activity's "events" array:
        {
          "event_type": "message",
          "name": "InitialWorkflowTrigger", // Or derived from diagram label
          "event_label": "initial_trigger_message_received", // Globally unique
          "next": "<id_of_the_first_actual_task_or_gateway>" // e.g., "activity_1"
          // "trigger_identifier" and "event_definition" as appropriate
        }
        ```
*   *   The `next` field within this event entry will point to the subsequent element in the workflow.
*   *   Its outgoing flow is defined by the `next` field(s) in its `events` array. **Step 3 will use these `next` fields to generate corresponding `edge` records.**
*   *   **Do NOT generate a `start_end_node` (in Step 2G) for such an element if it has been converted to an activity here.**"

**Output:** One JSON Line per activity.

*   `__type`: `"activity"`
*   `id`: Unique ID (snake_case, e.g., `validate_order`, `wait_for_approval`). Must be unique across all elements.
*   `type`: The nature of the activity.
    *   `"task"`: A unit of work performed. Requires `task_type`.
    *   `"wait_for_event"`: A pause state waiting for one or more specific events (message, timer, signal, etc.) to occur. Requires the `events` array.
*   `task_type`: **(ONLY for `type: "task"`)** Specifies the kind of task. **Mandatory if `type` is `"task"`**. Further specialization based on BPMN notation icons or other clear indicators:
    *   `"user_task"`: Activities performed by humans (indicated by person icon)
    *   `"service_task"`: Automated activities/API calls (indicated by gear icon)
    *   `"send_task"`: Activities that send messages (indicated by envelope/arrow icon)
    *   `"receive_task"`: Activities that receive messages (indicated by incoming message icon)
    *   `"script_task"`: Activities that execute scripts (indicated by script/code icon)
    *   `"business_rule_task"`: Activities applying business rules (indicated by table/rule icon)
    *   `"manual_task"`: Non-automated human tasks (indicated by hand icon)
    *   `"call_activity"`: Activities that call other processes (indicated by bold border with plus sign)
    *   `"standard_task"`: Default when no specialized icon is present
*   `participant`: ID of the participant performing the activity (from Step 2A).
*   `description`: Concise, action-oriented description. Reflect ONLY what's visible or clearly implied. E.g., "Validates customer credit score", "Waits for manager's decision or timer expiry".
*   `data_objects_flow`: Array of interactions with data objects (identified in Step 2D). Each object: `{ "id": "<data_object_id>", "data_flow": "<input|output|read|update>" }`.
*   `events`: **(ONLY for `type: "wait_for_event"`) Mandatory.** Array describing the event(s) this activity waits for. The activity proceeds along the path (`next`) of the *first* event that occurs.
    *   Each object in the array represents one potential triggering event and **MUST** contain:
        *   `event_type`: **Mandatory.** The type of event, e.g., `"message"`, `"timer"`, `"signal"`, `"conditional"`, or `"participant_interaction"`.
        *   `name`: **Mandatory.** A descriptive name for the event branch (e.g., `"ApprovalReceived"`, `"TimerExpired"`, `"SignalReceived"`).
        *   `next`: **Mandatory.** The `id` of the next element (activity/gateway) activated when *this specific event* occurs.
        *   `event_definition`: **(Required for certain `event_type`s)** Provides specific details for the event trigger:
            *   For `event_type: "timer"`: **Required.** Specifies the timer condition. Can be:
                *   An ISO 8601 Duration string (e.g., `"P3D"` for 3 days, `"PT1H"` for 1 hour). Indicates waiting for a duration relative to when the activity started.
                *   An ISO 8601 DateTime string (e.g., `"2024-12-31T23:59:59Z"`). Indicates waiting until a specific date/time.
                *   A Cron expression string (e.g., `"0 18 * * 5"` for every Friday at 6 PM). Indicates waiting for the next occurrence matching the schedule (in this case of the cron example `"0 18 * * 5"`, next coming Friday at 6).
                *   A descriptive string (e.g., `"Friday 6 pm"`) if the exact format cannot be determined. *If using a descriptive string, also add an `unrecognized_item` (Step 2F) with `category: "ambiguous_timer_definition"`.*
            *   For `event_type: "message"`: Message name or correlation info (e.g., `"OrderConfirmation_123"`). Often omitted if implicit.
            *   For `event_type: "signal"`: Signal name (e.g., `"GlobalShutdown"`).
            *   For `event_type: "conditional"`: Condition expression.
            *   For `event_type: "participant_interaction"`: Indicates a wait for a specific participant's action (e.g., `"UserActionRequired"`).
        *   `event_label`: **Mandatory. Globally Unique, Variable-Friendly Identifier.** A short, descriptive label in `snake_case` suitable for use as a variable, enum value, or function name component in generated code.
            *   **Purpose:** Represents the specific *outcome* or *state* resulting from this event occurring. Answers "What happened?".
            *   **Uniqueness:** **MUST be globally unique across ALL `event_label`s AND `condition_label`s generated in the entire IR.**
            *   **Content:** Derived from the `name` and/or `event_type`, potentially incorporating context from the activity's `id` or `description` to ensure global uniqueness and clarity. Should represent the outcome (e.g., `approval_received`, `timeout_expired`, `update_message_processed`, `cancellation_signal_detected`, `daily_timer_triggered`).
            *   **Examples (Good - Context  Outcome):**
                *   For activity `wait_for_approval_or_timeout` event `name: "ApprovalReceived"`: `approval_received` (if unique) or `manager_approval_received` (if needed for uniqueness).
                *   For activity `wait_for_approval_or_timeout` event `name: "Timeout"`: `approval_timeout_expired`.
                *   For activity `wait_external_update` event `name: "UpdateReceived"`: `external_update_received`.
            *   **Examples (Bad - Avoid These):**
                *   `event_1`, `event_a` (Not descriptive)
                *   `message`, `timer` (Too generic, use the outcome)
                *   `true`, `false` (Not applicable/descriptive for events)
                *   `wait_for_approval_or_timeout` (This is the activity ID)
            *   **Constraints (MUST follow):**
                *   Must be valid `snake_case`.
                *   Must be globally unique across all `event_label`s.
                *   Must represent the specific event outcome.
                *   MUST NOT be identical to the activity's `id`.
            *   **(Mandatory)**
        *   `trigger_identifier`: **(Optional but recommended for Message/Signal)** The specific name/identifier of the message or signal being waited for (e.g., `"PaymentReceivedMessage"`, `"UserCancelSignal"`). Can overlap with `name`.
*   `asynchronous`: Always `true` for `type: "wait_for_event"`. For `type: "task"`, set `true` if likely involves waiting or non-immediate execution, `false` otherwise (default).
*   `multi_instance_type`: **(Optional)** If standard BPMN multi-instance markers are clearly visible on the activity shape: `"parallel"` (for `|||`) or `"sequential"` (for `☰`). Omit or set to `null` if not present.
*   `confidence`: `"high"`, `"medium"`, or `"low"`.

**How Indefinite Waits are Represented:**
An activity waits indefinitely for non-timer events (like messages or signals) if its `events` array simply **does not contain any entry** with `event_type: "timer"`.

**Constraint Checks & Handling:**
*   `task_type` MUST be present if and only if `type` is `"task"`.
*   `events` array MUST be present and non-empty if and only if `type` is `"wait_for_event"`.
*   Entries in `events` MUST have `event_type`, `name`, and `next`. `event_definition` is mandatory for timers.
*   Check for missing failure paths for `wait_for_event` (if description implies failure but no failure event listed), generating `unrecognized_item` (Step 2F) and setting `feedback_required: true` in metadata if needed.

```json
// Example Task (User)
{"__type": "activity", "id": "review_application", "type": "task", "task_type": "user_task", "participant": "p3_manager", "description": "Manager reviews the submitted application.", "data_objects_flow": [{"id": "application_doc", "data_flow": "read"}], "asynchronous": false, "multi_instance_type": null, "confidence": "high"}

// Example Wait for Event (Message OR Duration Timer) - Using trigger_identifier and event_label
{"__type": "activity", "id": "wait_for_approval_or_timeout", "type": "wait_for_event", "participant": "p2_order_system", "description": "Waits for manager approval message or times out after 2 days.", "data_objects_flow": [], "events": [{"event_type": "message", "name": "Approval Received", "event_label": "manager_approval_received", "trigger_identifier": "ManagerApprovalMsg", "next": "process_approved_order"}, {"event_type": "timer", "name": "Timeout Expired", "event_label": "approval_timeout_expired", "trigger_identifier": null, "next": "escalate_lack_of_approval", "event_definition": "P2D"}], "asynchronous": true, "multi_instance_type": null, "confidence": "high"}

## 📌 Step 2C: Extract *Explicit* Gateways

**Output Progress Update:**
```json
{"__type": "system", "__step": "2.C", "__description": "Extract Branching and Decision Points"}
```

**Task:** Identify ONLY visually represented gateways (typically diamonds) used for branching, merging, or event-based decisions. **Do NOT infer gateways not explicitly drawn.** Determine the gateway `type` based on internal symbols and the `role` based primarily on sequence flow patterns.

**Identify Type Based on Internal Symbol:**

*   **"X"** marker: **Exclusive (`type: "exclusive"`)**.
    *   **Logic:** *Exactly ONE* outgoing path is chosen based on conditions evaluating to true (or the default path is taken). Think `if/else if/else`.
    *   **Visual:** A diamond shape containing an 'X' marker or no marker (plain diamond, usually assumed exclusive *unless role clearly indicates converging*).

*   **"+"** marker: **Parallel (`type: "parallel"`)**.
    *   **Logic:** All outgoing paths are activated *simultaneously*. Conditions are NOT used on outgoing paths for diverging parallel gateways.
    *   **Visual:** A diamond shape containing a '+' marker.

*   **"O"** marker (or Circle): **Inclusive (`type: "inclusive"`)**.
    *   **Logic:** *One OR MORE* outgoing paths are chosen. For *each* outgoing path, its condition is evaluated. *ALL* paths whose conditions evaluate to true are activated concurrently. If no conditions evaluate to true and a default path exists, the default path is taken.
    *   **Visual:** **A diamond shape containing an 'O' (circle) marker. This 'O' symbol is the PRIMARY indicator for an inclusive gateway.** Do not confuse it with Exclusive just because it has conditions on its outgoing paths.

*   **Star/Pentagon/Event Icon**: **Event-Based (`type: "event-based"`)**.
    *   **Logic:** Waits for the *first* of several possible events to occur, then takes *only* the corresponding path.
    *   **Visual:** A diamond shape containing a specific event marker (like a star, pentagon, message symbol, etc.).

*   **No Symbol (Plain Diamond)**: **Requires careful analysis based on Role and Outgoing Paths:**
    1.  **Determine `role`** (diverging/converging) based on incoming/outgoing edge count first.
    2.  **If `role` is `converging`**: Assume `type: "exclusive"` as the default merge type. (Downstream logic might refine this if needed).
    3.  **If `role` is `diverging`**:
        *   **Examine the outgoing sequence flow arrows:** Look for any text labels directly on or immediately next to these arrows that represent **conditions** (e.g., "Yes", "No", ">100", "Type A", "Approved", "Valid", "Failed").
        *   **If conditions ARE visible on outgoing paths:** Assign **`type: "exclusive"`**. The presence of conditions implies a choice, making exclusive the most likely intent for a plain diamond. Proceed to populate the `flows` array based on these conditions.
        *   **If conditions ARE NOT visible on outgoing paths:** Assign **`type: "parallel"`**. The absence of conditions strongly suggests the intent was for unconditional, concurrent paths. Ensure `flows` remain empty/omitted for this gateway.
        *   **(Self-Correction Note:** This requires the LLM to visually assess the outgoing paths for conditional labels during Step 2C, *before* Step 3 formally generates edges and their `visual_label` attribute.)

**Hints for Differentiating Exclusive ('X'/Plain) vs. Inclusive ('O') Diverging Gateways:**

While the **internal symbol ('O' for Inclusive, 'X'/Plain for Exclusive) is the most important factor**, consider these secondary clues if the symbol is ambiguous or for confirmation:

1.  **Nature of Conditions (Analyze outgoing path labels/logic):**
    *   **Suggests Exclusive:** Conditions often represent choices that seem **mutually exclusive (only one path can be logically true at a time)**. Examples:
        *   Selecting one status from several (`'Approved'` vs. `'Rejected'` vs. `'Needs Info'`).
        *   Choosing one category from a set (`'Type A'` vs. `'Type B'` vs. `'Type C'`).
        *   Evaluating against distinct, non-overlapping ranges that cover all possibilities (`<100` vs. `>=100 and <200` vs. `>=200`).
        *   Simple Yes/No answers to a single, clear question (`'Is Valid?' Yes/No`).
    *   **Suggests Inclusive:** Conditions might represent **independent options or features (zero, one, or potentially multiple paths could be logically true simultaneously)**. Examples:
        *   Selecting optional features (`'Send Email?'`, `'Include Gift Wrap?'`, `'Expedite Shipping?'` - where multiple could be true).
        *   Triggering parallel checks based on different criteria (`'Credit Score Low?'`, `'Inventory Low?'` - where both could be true).
        *   Conditions that don't inherently exclude each other (`'Is VIP Customer?'`, `'Order Value > $500?'`).
    *   *(Caveat: This is only a hint; rely on the 'O' symbol if present for Inclusive.)*

2.  **Gateway Question/Description (Analyze gateway's own label/purpose):**
    *   **Suggests Exclusive:** The question often implies selecting a single outcome, state, or classification (e.g., "Determine Order Status", "Select Payment Method", "Check Validation Result").
    *   **Suggests Inclusive:** The question might imply selecting multiple applicable options or checking several independent conditions (e.g., "Select Notification Options", "Check Applicable Flags", "Evaluate Optional Services").

**Remember: If a diamond gateway has an 'O' (circle) marker inside, classify it as `type: "inclusive"` regardless of other hints. If it has an 'X' or is plain, classify it as `type: "exclusive"` (assuming it's diverging).**

**Determining Gateway Role (`role`) - PRIORITIZE THIS:**

*   Analyze the expected number of incoming and outgoing **sequence flow edges** connected to the gateway (based on visual inspection, formal edges defined in Step 3).
*   **`diverging`**: **MUST** have **ONE** incoming edge and **MORE THAN ONE** outgoing edge. The `flows` array will describe the conditions or events for these outgoing paths.
*   **`converging`**: **MUST** have **MORE THAN ONE** incoming edge and **EXACTLY ONE** outgoing edge. The `flows` array for converging gateways **MUST** be empty (`[]`).
*   **`dual`**: (Rare for single explicit gateways) Acts as both. If encountered, analyze context carefully. If ambiguous, consider if it should be modeled as separate diverging/converging elements or flag as `unrecognized_item` (Step 2F) with `category: "ambiguous_gateway_role"`.
*   **Rule for Plain Diamonds:** The `role` for plain diamonds **MUST** be determined by the incoming/outgoing edge count as defined above. Do NOT assume diverging if it clearly acts as a merge point.

*(Self-Correction Note: While edges are formally generated in Step 3, accurately determining the `role` (converging vs. diverging) requires mentally tracing or predicting the incoming and outgoing sequence flows based on the diagram during this step.)*
