# Determinism Checklist — Python

Apply each rule below to every file in `workflow_files` produced by `review-detect-target.md`. The "scope" of every rule is the body of any function decorated with `@wfr.workflow(...)` (or any helper called from such a function). Findings inside `@wfr.activity(...)` functions are out of scope for this skill.

Rule source: see [`../create-workflow-python/REFERENCE.md`](../create-workflow-python/REFERENCE.md) — section "Workflow determinism".

| Rule id     | Severity | What to detect (regex / pattern in workflow scope)                                                                       | Why it fails                                                                                              | Suggested fix                                                                              |
| ----------- | -------- | ------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| DWF-DET-001 | critical | `datetime\.now\(`, `datetime\.utcnow\(`, `date\.today\(`, `time\.time\(`, `time\.monotonic\(`                            | Wall-clock reads diverge between original and replay; values differ each run.                             | Use `ctx.current_utc_datetime`.                                                            |
| DWF-DET-002 | critical | `uuid\.uuid[14]\(`, `secrets\.token_`                                                                                     | Random / per-run identifiers diverge across replays.                                                      | Generate ids inside an activity, or derive from `ctx.instance_id` deterministically.       |
| DWF-DET-003 | critical | `random\.`, `numpy\.random\.`, `secrets\.choice`, `os\.urandom`                                                          | Random sources produce different values per replay.                                                       | Move randomness into an activity and return the value.                                     |
| DWF-DET-004 | critical | `time\.sleep\(`, `await asyncio\.sleep\(`                                                                                 | Blocks the workflow loop and is not replay-aware.                                                         | Use `yield ctx.create_timer(timedelta(...))`.                                              |
| DWF-DET-005 | critical | `asyncio\.create_task\(`, `asyncio\.gather\(`, `loop\.run_in_executor\(`, `threading\.Thread\(`, `multiprocessing\.`     | Escapes the workflow scheduler; tasks are invisible to history and may run twice on replay.               | Schedule activities with `ctx.call_activity(...)` and combine via `wf.when_all(...)`.      |
| DWF-DET-006 | critical | `requests\.`, `httpx\.`, `urllib`, `aiohttp\.`, `socket\.`                                                               | Direct network I/O inside a workflow is non-deterministic and runs once per replay.                       | Wrap the call in an activity.                                                              |
| DWF-DET-007 | critical | `open\(`, `pathlib\.Path\(.*\)\.(read|write)`, `shutil\.`, `os\.remove`, `os\.rename`                                    | File I/O is non-deterministic and runs once per replay.                                                   | Wrap the call in an activity.                                                              |
| DWF-DET-008 | critical | DB clients in workflow scope: `psycopg`, `sqlalchemy`, `pymongo`, `redis\.`, `cosmos`, `boto3\.client\(`, `azure\.`     | Database calls are non-deterministic and side-effecting.                                                  | Wrap the call in an activity.                                                              |
| DWF-DET-009 | critical | `os\.environ`, `os\.getenv\(`, `Settings\(\)`, `dotenv\.`                                                                 | Configuration values can change between original run and replay.                                          | Pass into the workflow input or read inside an activity.                                   |
| DWF-DET-010 | warning  | `for .* in (dict|set)\(` driving `ctx.call_activity(...)` order                                                          | `dict`/`set` iteration order in the workflow body is risky if the source is built non-deterministically.  | Sort before iterating, or pass an ordered list into the workflow.                          |
| DWF-DET-011 | warning  | Module-level mutable globals (`global ...`, list/dict at module top) read or written from workflow scope                  | State persists across replays and across instances; mutations cause divergence.                           | Move state into the workflow input/output, or into an activity.                            |
| DWF-DET-012 | warning  | `print(`, `logging\.` calls in workflow scope                                                                             | Printed/logged once per replay, multiplying log volume and confusing diagnostics.                          | Emit logs from inside activities, or guard with `if not ctx.is_replaying:`.                |
| DWF-DET-013 | critical | `while True:` or `while <cond>:` without a `ctx.continue_as_new(` inside                                                  | Eternal loops grow workflow history without bound.                                                        | Restructure as `ctx.continue_as_new(...)` after each iteration.                            |
| DWF-DET-014 | warning  | `async def` workflow function with `await` on non-`ctx.*` awaitables                                                      | Awaiting non-workflow awaitables blocks outside the workflow scheduler.                                   | Only `yield ctx.call_activity(...)` / `ctx.create_timer(...)` / `wf.when_all(...)` etc.    |
| DWF-DET-015 | info     | `time.perf_counter`, `time.process_time`                                                                                  | Timing measurements drift between original and replay.                                                    | Capture timestamps with `ctx.current_utc_datetime` at boundaries instead.                  |

## Confidence notes

- DWF-DET-005 — `asyncio.gather` is a common Python idiom; flag only inside workflow scope, since gather inside an activity is fine.
- DWF-DET-008 — Many DB symbols may appear only as imports for type hints; only flag when actually invoked (parentheses or attribute access on an instance).
- DWF-DET-012 — Output via `print` / `logging` is occasionally tolerated for early-stage debugging; downgrade to "Please verify" if the call is guarded by `if not ctx.is_replaying:`.

## Cross-reference

If a rule trips inside a helper function, confirm the helper is reachable from a `@wfr.workflow`-decorated function before reporting it. Pure helpers that the workflow never calls are out of scope.
