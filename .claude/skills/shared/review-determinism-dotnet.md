# Determinism Checklist — .NET

Apply each rule below to every file in `workflow_files` produced by `review-detect-target.md`. The "scope" of every rule is the body of any method on a class deriving from `Workflow<TInput, TOutput>` — typically `RunAsync`, plus any private helper called from it. Findings inside `WorkflowActivity<>` classes are out of scope for this skill.

Rule source: see [`dotnet-workflow-class.md`](dotnet-workflow-class.md) — section "Workflow determinism".

| Rule id     | Severity | What to detect (regex / pattern in workflow scope)                                                                  | Why it fails                                                                                                          | Suggested fix                                                                  |
| ----------- | -------- | ------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| DWF-DET-001 | critical | `DateTime\.(Now|UtcNow|Today)` or `DateTimeOffset\.(Now|UtcNow)`                                                    | Wall-clock reads diverge between original run and replay, so awaited tasks resolve to different values on replay.     | Use `context.CurrentUtcDateTime`.                                              |
| DWF-DET-002 | critical | `Guid\.NewGuid\(\)`                                                                                                  | Generates a fresh value per replay, breaking deterministic history matching.                                          | Use `context.NewGuid()`.                                                       |
| DWF-DET-003 | critical | `new Random\(`, `Random\.Shared`, `RandomNumberGenerator\.`                                                          | Random sources produce different values per replay.                                                                   | Generate randomness inside an activity and return it as the activity output.   |
| DWF-DET-004 | critical | `Thread\.Sleep\(`, `Task\.Delay\(`                                                                                   | Blocks the workflow thread and is not replay-aware.                                                                   | Use `await context.CreateTimer(...)`.                                          |
| DWF-DET-005 | critical | `Task\.Run\(`, `Task\.Factory\.StartNew\(`, `new Thread\(`, `ThreadPool\.`                                           | Escapes the workflow scheduler — the resulting task is invisible to history and may run twice on replay.              | Move the work into an activity called via `context.CallActivityAsync(...)`.    |
| DWF-DET-006 | critical | `HttpClient`, `HttpRequestMessage`, `WebClient`, `Socket`, `TcpClient`, `SmtpClient`                                 | Direct network I/O inside a workflow is non-deterministic and runs once per replay.                                   | Wrap the call in an activity.                                                  |
| DWF-DET-007 | critical | `File\.`, `Directory\.`, `FileStream`, `Path\.GetTempFileName`, `StreamReader\(`, `StreamWriter\(`                   | File I/O is non-deterministic and runs once per replay.                                                               | Wrap the call in an activity.                                                  |
| DWF-DET-008 | critical | DB clients in workflow scope: `SqlConnection`, `DbContext`, `MongoClient`, `Cosmos`, `Redis`, `IDbConnection`        | Database calls are non-deterministic and side-effecting; results differ between original and replay.                  | Wrap the call in an activity.                                                  |
| DWF-DET-009 | critical | `Environment\.GetEnvironmentVariable`, `IConfiguration\[`, `IConfiguration\.GetSection`, `Environment\.MachineName`  | Configuration values can change between original run and replay (e.g., after a redeploy).                             | Pass the value into the workflow input or read it inside an activity.          |
| DWF-DET-010 | warning  | `foreach.*\b(Dictionary|HashSet|ConcurrentDictionary)<` used to drive `CallActivityAsync` order                      | Hash-based iteration order is not guaranteed stable across runtimes.                                                  | Sort the keys before iterating, or pass an ordered list (`List<>`/`Array`).    |
| DWF-DET-011 | warning  | `static` fields/properties read or written in workflow scope                                                         | Captured outer state lives across replays and across instances; mutations cause divergence.                           | Move state into the workflow input/output, or into an activity.                |
| DWF-DET-012 | warning  | `Console\.WriteLine`, `Trace\.WriteLine`, `ILogger` *not* obtained via `context.CreateReplaySafeLogger`              | Logs are emitted once per replay, multiplying log volume and confusing diagnostics.                                   | Replace with `var logger = context.CreateReplaySafeLogger<TWorkflow>();`.      |
| DWF-DET-013 | critical | `while \(.*\)` or `for \(;;\)` (unbounded loops) in workflow scope without a `context.ContinueAsNew(` inside         | Eternal loops grow workflow history without bound and eventually exceed actor state limits.                           | Restructure as `ContinueAsNew(...)` after each iteration.                      |
| DWF-DET-014 | warning  | `async void` methods on a `Workflow<>` class                                                                         | `async void` cannot be awaited and bypasses the workflow scheduler.                                                   | Return `Task` / `Task<T>` instead.                                             |
| DWF-DET-015 | info     | `Stopwatch\.StartNew`, `Stopwatch\.GetTimestamp`                                                                     | Timing measurements drift between original run and replay.                                                            | Capture timestamps with `context.CurrentUtcDateTime` at boundaries instead.    |

## Confidence notes

- DWF-DET-005 — `Task.Run` may legitimately appear inside a `partial` test helper that lives in the same file; check the enclosing class is the workflow before flagging as critical, otherwise emit as "Please verify".
- DWF-DET-008 — Many DB types are also legitimate type names in workflow input/output records; only flag when the symbol is used as a constructor or method receiver in workflow scope.
- DWF-DET-012 — `ILogger` is fine if it was obtained from `context.CreateReplaySafeLogger`; verify by tracing the variable's origin in the same method.

## Cross-reference

If a rule trips inside a method that is *not* `RunAsync`, confirm the method is reachable from `RunAsync` before reporting it. Pure helper methods that the workflow never calls are out of scope.
