# Agent Tools Checklist — Python

Apply each rule below to every file in `tool_files` produced by `review-detect-target-agent.md`. The "scope" of every rule is a function decorated with `@tool` (or a framework-specific tool decorator such as `function_tool`).

Rule source: see [`../create-agent-python/REFERENCE.md`](../create-agent-python/REFERENCE.md) — section "Tools".

| Rule id      | Severity | What to detect                                                                                                     | Why it matters                                                                                           | Suggested fix                                                                              |
| ------------ | -------- | ------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| DAG-TOOL-001 | critical | Tool performs a non-idempotent mutation (`requests.post`, DB insert, file write) without an idempotency key        | Dapr Workflow may retry the activity; without an idempotency key the side effect runs more than once.  | Accept an `idempotency_key` arg (or read `taskExecutionId`) and use it on the outbound call.|
| DAG-TOOL-002 | warning  | Tool function has no docstring, or the docstring is one word / empty                                               | The LLM routes tool calls by docstring; a vague docstring silently degrades agent behaviour.            | Write a one-line sentence describing what the tool does and when to call it.                |
| DAG-TOOL-003 | warning  | Tool returns an unbounded payload — `return response.json()` or `return rows` with no size guard                   | Large returns bloat the agent context and can exhaust the LLM token budget.                              | Truncate, summarize, or return a reference id the agent can fetch selectively.               |
| DAG-TOOL-004 | warning  | Tool has `except Exception: return "..."` / `except: pass` (swallowed exception)                                   | Swallowing masks retries and turns failures into silent successes the LLM cannot react to.              | Re-raise, or return a typed `ToolError` pydantic model the LLM can inspect.                 |
| DAG-TOOL-005 | info     | Tool function parameters are not type-hinted                                                                       | The framework-generated JSON schema is weaker; arg validation degrades.                                  | Add Python type hints for every parameter and a `Field(description=...)` when possible.      |
| DAG-TOOL-006 | warning  | Tool has no `args_model` Pydantic class and accepts free-form `**kwargs`                                            | Inputs are unvalidated; malformed tool calls reach the function body instead of being rejected upfront.  | Define a `BaseModel` subclass and pass it via `@tool(args_model=...)`.                      |
| DAG-TOOL-007 | warning  | Tool performs a synchronous HTTP call (`requests.get/post`) inside an `async def` tool                              | Blocks the agent's event loop and serialises I/O.                                                       | Use `httpx.AsyncClient` or wrap the call in `asyncio.to_thread`.                            |
| DAG-TOOL-008 | info     | Tool name does not match snake_case or does not clearly express a verb (`data`, `x`, `foo`)                         | Poor naming degrades LLM tool selection.                                                                | Rename to `<verb>_<noun>` (e.g. `get_weather`, `send_invitation`).                         |
| DAG-TOOL-009 | warning  | `print(` used inside a tool instead of `logging`                                                                    | Bypasses structured logging and loses trace correlation.                                                 | Use `logging.getLogger(__name__).info(...)`.                                                |
| DAG-TOOL-010 | warning  | Tool reads or writes module-level mutable globals                                                                   | Tools run on shared workers; globals leak across agent invocations.                                     | Pass state explicitly, inject dependencies, or store state via a Dapr state component.       |
| DAG-TOOL-011 | critical | Tool constructs an outbound URL (`httpx.`, `requests.`, `urlopen`) from an agent-supplied argument with no allowlist | Server-side request forgery: the LLM can direct the tool at internal services, cloud metadata endpoints (`169.254.169.254`), or loopback. | Validate the host against an allowlist before the call; block private / link-local / loopback IPs; disable redirect-following. |
| DAG-TOOL-012 | critical | Tool uses an agent-supplied argument as a file path (`open`, `pathlib.Path`, `os.path.join`) without a root-directory check | Path traversal: agent input `../../../etc/passwd` escapes the intended directory. | Resolve the path against an allowed root and assert the resolved path is still inside the root. Reject symlinks that resolve outside. |
| DAG-TOOL-013 | warning  | Tool returns externally-sourced content (HTTP response body, DB row, user-controlled store) directly to the agent | Prompt injection: adversarial content in the response can override the agent's instructions on the next turn. | Sanitise, truncate, or wrap the content in a typed Pydantic model; strip instruction-like prefixes (e.g. `"IGNORE PREVIOUS..."`). |

## Confidence notes

- DAG-TOOL-001 — Idempotency may appear as an `Idempotency-Key` header, a UUID passed as an argument, or the framework's `taskExecutionId`. Treat as satisfied if any of these are visible on the outbound call.
- DAG-TOOL-007 — Only flag inside `async def` bodies. Synchronous tools are allowed.

## Cross-reference

Rules about memory and state configuration are not in scope here — see `review-agent-memory-python.md`. Rules about pub/sub topic conventions for multi-agent setups are in `review-agent-orchestration-python.md`.
